Nếu count < 5 → cho phép tiếp tục; tăng bộ đếm: INCR ip_rate:{IP}, EXPIRE 900s.Server-side Validator Middleware (express-validator) kiểm tra toàn bộ trường đầu vào theo quy tắc nghiệp vụ.Auth Controller truy vấn MongoDB: User.findOne({ email }) — kiểm tra email đã tồn tại chưa.Nếu email chưa tồn tại: bcrypt.hash(password, 10) để mã hoá mật khẩu.Tạo bản ghi người dùng mới trong MongoDB với trạng thái status = 'PENDING': INSERT user (fullName, email, passwordHash, phoneNumber, apartmentNumber, status='PENDING').Sinh mã OTP 6 chữ số: crypto.randomInt(100000, 999999) = ví dụ 847291.Lưu hash OTP vào Redis: SET otp:{userId} bcrypt(OTP) EX 300 (TTL 5 phút; one-time use).Gửi email OTP qua SMTP Server: SendMail({ to: email, subject: '[ApartmentHub] Mã xác thực OTP', html_body }).API trả về HTTP 202 Accepted: { success: true, message: 'OTP đã được gửi đến email', data: { userId, email (masked), otpExpiresAt } }.ReactJS chuyển sang màn hình nhập OTP, hiển thị đồng hồ đếm ngược 5 phút.Người dùng nhập mã OTP nhận được trong email và nhấn 'Xác nhận'.Frontend gửi POST /api/v1/auth/verify-otp { userId, otp }.Server lấy hash OTP từ Redis: GET otp:{userId}. Kiểm tra TTL còn hạn.bcrypt.compare(otp, hashOTP) = true — OTP hợp lệ.Xóa OTP khỏi Redis: DEL otp:{userId} (đảm bảo one-time use).Cập nhật trạng thái tài khoản trong MongoDB: UPDATE user SET status='ACTIVE' WHERE id={userId}.Sinh JWT Access Token + Refresh Token với payload { userId, email, role }.Trả về HTTP 200: { token, refreshToken, user: {id, name, role} } với HttpOnly Cookie.ReactJS redirect người dùng đến /dashboard — Đăng ký và kích hoạt tài khoản thành công.2.4 Luồng Sự Kiện Thay Thế (Alternative Flows)2.4.1 Client-side Validation Thất BạiEmail sai định dạng → hiển thị lỗi inline: 'Email không hợp lệ', không gửi request.Mật khẩu không đủ 8 ký tự hoặc thiếu chữ hoa/số/ký tự đặc biệt → thông báo yêu cầu độ phức tạp.confirmPassword không trùng password → 'Mật khẩu xác nhận không khớp'.Số điện thoại sai format VN (không phải 10 chữ số bắt đầu bằng 0) → báo lỗi định dạng.Trường bắt buộc bỏ trống → highlight viền đỏ và thông báo 'Trường này là bắt buộc'.2.4.2 Rate Limiting Bị Kích Hoạt (HTTP 429)Người dùng (hoặc bot) gửi request đăng ký lần thứ 6+ trong 15 phút từ cùng một IP.Rate Limiter phát hiện count = 5, TTL còn 720s → chặn ngay, không chuyển sang controller.Ghi nhật ký bảo mật: LOG RateLimitExceeded { IP, timestamp, userAgent }.Trả về HTTP 429 Too Many Requests với header Retry-After: 720 và X-RateLimit-*.ReactJS hiển thị: 'Quá nhiều yêu cầu. Vui lòng thử lại sau 12 phút', đếm ngược trực tiếp.2.4.3 Email Đã Tồn Tại (HTTP 409)Controller truy vấn MongoDB tìm thấy user với email đã nhập.Không tạo tài khoản mới; không gửi email.Trả về HTTP 409 Conflict: { error: 'Email đã được sử dụng. Vui lòng đăng nhập hoặc dùng email khác.' }.ReactJS hiển thị thông báo lỗi inline tại trường Email.2.4.4 Server-side Validation Thất Bại (HTTP 422)express-validator phát hiện vi phạm quy tắc nghiệp vụ (dù client đã validate).Trả về HTTP 422 Unprocessable Entity: { errors: [{ field, message }] } — danh sách lỗi chi tiết từng trường.ReactJS ánh xạ lỗi và hiển thị bên dưới trường tương ứng.2.4.5 OTP Sai (HTTP 400)bcrypt.compare(otp, hash) = false → OTP không khớp.INCR otp_attempts:{userId} → đếm số lần sai.Trả về HTTP 400: { error: 'OTP không hợp lệ', remaining: X lần còn lại }.Sau 3 lần sai: DEL otp:{userId}, DELETE user PENDING → HTTP 403, yêu cầu đăng ký lại.2.4.6 OTP Hết HạnTTL Redis của key otp:{userId} đã về 0 — key không còn tồn tại.Server trả về HTTP 400: { error: 'OTP đã hết hạn. Vui lòng yêu cầu gửi lại.' }.ReactJS hiển thị nút 'Gửi lại OTP' (nếu chưa hết lượt gửi lại).2.4.7 Gửi Lại OTP (Resend OTP)Người dùng nhấn 'Gửi lại OTP' sau 60 giây cooldown.Server kiểm tra Redis: resend_cooldown:{userId} = null → cho phép.Kiểm tra otp_resend_count:{userId} < 3 → cho phép tối đa 3 lần gửi lại.Vô hiệu hóa OTP cũ, sinh OTP mới, cập nhật Redis TTL 5 phút, gửi email lại.Đặt cooldown mới: SET resend_cooldown:{userId} EX 60.3. Authorization Middleware cần đảm bảo người dùng chỉ được sửa hồ sơ của chính mình. Đây là nguyên tắc xác thực và phân quyền quan trọng trong JWT .Luồng sự kiện chính (Main Flow)Bước 1:Người dùng nhấn vào menu “Hồ sơ” hoặc biểu tượng avatar cá nhân.Bước 2:Hệ thống kiểm tra JWT token trong cookie hoặc Authorization Header thông qua Auth Middleware.Bước 3:JWT Middleware xác thực token hợp lệ và giải mã thông tin userId từ token .Bước 4:Hệ thống truy xuất thông tin hồ sơ hiện tại từ database theo userId.Bước 5:Hệ thống hiển thị trang Profile với form đã điền sẵn thông tin hiện tại.Bước 6:Người dùng chỉnh sửa các trường như họ tên, số điện thoại, địa chỉ và có thể chọn ảnh avatar mới.Bước 7:Người dùng nhấn nút “Lưu thay đổi”.Bước 8:Frontend thực hiện validate dữ liệu cơ bản như định dạng số điện thoại, trường bắt buộc và dung lượng ảnh.Bước 9:Dữ liệu được gửi đến API backend bằng request PUT hoặc POST.Bước 10:Backend tiếp tục thực hiện Input Validation bằng express-validator để đảm bảo dữ liệu hợp lệ từ phía server .Bước 11:Authorization Middleware kiểm tra quyền truy cập, chỉ cho phép user chỉnh sửa profile của chính mình.Bước 12:Nếu có ảnh mới, Multer xử lý upload và lưu file vào hệ thống lưu trữ.Bước 13:Hệ thống cập nhật thông tin mới vào MongoDB.Bước 14:Hệ thống trả về response thành công cùng thông tin profile đã cập nhật.Bước 15:Frontend cập nhật lại giao diện và hiển thị thông báo “Cập nhật hồ sơ thành công”.Luồng thay thế (Alternative Flow)Trường hợp 1: Token hết hạnNếu JWT token đã hết hạn, Middleware trả về lỗi 401 Unauthorized. Frontend sẽ yêu cầu đăng nhập lại hoặc sử dụng Refresh Token nếu hệ thống có hỗ trợ.Trường hợp 2: Token không hợp lệNếu token không đúng hoặc bị giả mạo, hệ thống trả về thông báo “Không có quyền truy cập”.Trường hợp 3: Dữ liệu không hợp lệNếu số điện thoại sai định dạng, tên để trống hoặc dữ liệu không hợp lệ, hệ thống hiển thị lỗi tại trường tương ứng và không cho phép cập nhật.Trường hợp 4: File ảnh không hợp lệNếu file upload không phải ảnh hoặc vượt quá dung lượng cho phép, Multer trả về lỗi và hiển thị thông báo “File không hợp lệ hoặc vượt quá dung lượng cho phép”.Trường hợp 5: Lỗi DatabaseNếu MongoDB gặp lỗi kết nối hoặc timeout, hệ thống trả về lỗi server và thông báo “Có lỗi xảy ra, vui lòng thử lại sau”.Trường hợp 6: Không có thay đổiNếu người dùng nhấn “Lưu” nhưng không thay đổi bất kỳ thông tin nào, frontend sẽ phát hiện và hiển thị thông báo “Không có thông tin nào thay đổi”.2.2.
